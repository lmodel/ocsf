# Known Upstream LinkML Issues

This file tracks bugs in [linkml](https://github.com/linkml/linkml) and
[linkml-runtime](https://github.com/linkml/linkml-runtime) that affect the
Python datamodel generated from this OCSF schema. None of them are caused by
the OCSF schema itself — they manifest whenever a LinkML class uses
`slot_usage` to refine an inherited slot's range to a more specific enum,
which is a pattern OCSF uses extensively (`activity_id`, `category_uid`,
`class_uid`, etc.).

Versions confirmed affected: `linkml==1.10.0`, `linkml-runtime==1.10.0`
(Python 3.14).

## Symptoms

Running `pytest tests/test_data.py` against the generated `ocsf.datamodel.ocsf`
fails for the following test fixtures:

```
tests/data/valid/Authentication-success.yaml
tests/data/valid/DnsActivity-query.yaml
tests/data/valid/Finding-vulnerability.yaml
tests/data/valid/VulnerabilityFinding-cve.yaml
```

Each fails with a variation of:

```
TypeError: cannot use 'ocsf.datamodel.ocsf.<X>ActivityIdEnum' as a dict key
(unhashable type: '<X>ActivityIdEnum')
```

The data files themselves are valid against the LinkML schema (`linkml-validate`
accepts them and the JSON Schema generated by `gen-json-schema` accepts them).
The failure is purely in the Python dataclass `__post_init__` produced by
`gen-python`.

## Root cause — three compounding bugs

### Bug A — `SchemaLoader.process_slot_usages` leaves an orphan parent slot

File: `linkml/utils/schemaloader.py:840-911` (function `process_slot_usages`).

When a class declares `slot_usage` to narrow an inherited slot's range, the
loader synthesises a new mangled slot (e.g. `Authentication_activity_id`) and
intends to replace the bare slot name in `cls.slots`. The replacement check at
line 906 looks for `parent_slot.name` in `cls.slots`. For a deep inheritance
chain (e.g. `Object → ... → BaseEvent → IamEvent → Authentication`), the
proximal parent's slot was *itself* mangled at a higher level (it's
`BaseEvent_activity_id`, not `activity_id`), so the lookup misses and the bare
`activity_id` is never removed.

Reproduction:

```python
from linkml.utils.schemaloader import SchemaLoader
loader = SchemaLoader("src/ocsf/schema/ocsf.yaml")
loader.resolve()
cls = loader.schema.classes["Authentication"]
[s for s in cls.slots if "activity" in s]
# ['activity_id', 'Authentication_activity_id']  ← duplicate!
```

The schema ends up with two slot entries pointing at the same Python
attribute, with two different ranges (`ActivityIdEnum` and
`AuthenticationActivityIdEnum`).

### Bug B — `pythongen` emits one cast block per slot in `domain_slots`

File: `linkml/generators/pythongen.py:724-768` (function `gen_postinits`).

`gen_postinits` iterates `self.domain_slots(cls)` twice — once for required
slots and once for optional slots — emitting an `isinstance` + cast block for
each. Combined with Bug A, both `Authentication_activity_id` (required, range
`AuthenticationActivityIdEnum`) AND the orphan `activity_id` (optional, range
`ActivityIdEnum`) flow through, so two cast blocks end up in
`__post_init__` referring to the same Python attribute:

```python
# (block 1 — required, from slot_usage)
if not isinstance(self.activity_id, AuthenticationActivityIdEnum):
    self.activity_id = AuthenticationActivityIdEnum(self.activity_id)
# ...
# (block 2 — optional, from the orphan parent slot)
if self.activity_id is not None and not isinstance(self.activity_id, ActivityIdEnum):
    self.activity_id = ActivityIdEnum(self.activity_id)
```

After block 1, `self.activity_id` is an `AuthenticationActivityIdEnum`
instance. Block 2's `isinstance(..., ActivityIdEnum)` is `False` (subclass
enums don't inherit from the parent enum class), so it re-enters the cast.

### Bug C — `EnumDefinitionImpl.__init__` is not idempotent on enum instances

File: `linkml_runtime/utils/enumerations.py:33-58` (`EnumDefinitionImpl.__init__`)
and `:25-30` (`EnumDefinitionMeta.__contains__`).

`__init__` handles `PermissibleValue` and `Curie` arguments but not existing
`EnumDefinitionImpl` subclass instances; for those it falls through to
`key = code` and then `key not in self.__class__`. The metaclass
`__contains__` is `item in cls.__dict__`, which requires `item` to be
hashable. The dataclass-generated enum subclasses are unhashable
(`@dataclass` without `eq=False` / `frozen=True` produces `__eq__` but
no `__hash__`), so the membership check raises `TypeError`.

Reproduction:

```python
from ocsf.datamodel.ocsf import AuthenticationActivityIdEnum, ActivityIdEnum
inst = AuthenticationActivityIdEnum("LOGON")
ActivityIdEnum(inst)
# TypeError: cannot use 'ocsf.datamodel.ocsf.AuthenticationActivityIdEnum'
# as a dict key (unhashable type: 'AuthenticationActivityIdEnum')
```

The combination of Bugs B and C means: any class whose `slot_usage` refines a
slot's range to a subclass enum produces a `__post_init__` that crashes the
moment that slot is populated.

## Suggested upstream fixes

- **Bug A** — In `process_slot_usages`, also probe for the *root* slot name
  (and any intermediate mangled names from ancestors) when deciding whether
  to replace an entry in `cls.slots`. The simplest patch walks the
  `is_a` chain of the parent slot and tries each name.
- **Bug B** — Once Bug A is fixed, `domain_slots` no longer returns the
  duplicate; the second cast block disappears. As a defence-in-depth measure,
  `gen_postinits` could de-duplicate by aliased Python field name before
  emitting.
- **Bug C** — `EnumDefinitionImpl.__init__` should treat an existing
  `EnumDefinitionImpl` instance as a no-op fast path (or extract its
  `_code.text` and re-resolve), so generated code is robust to the duplicate
  cast pattern even before Bugs A/B are fixed.

## Local workaround sketch

Until the upstream fixes land, a post-codegen script can repair the produced
`src/ocsf/datamodel/ocsf.py` by removing the redundant second-pass cast block.
The pattern to remove is, for any subclass enum `<X>ActivityIdEnum` that was
already cast earlier in the same method:

```python
        if self.activity_id is not None and not isinstance(self.activity_id, ActivityIdEnum):
            self.activity_id = ActivityIdEnum(self.activity_id)
```

Hook it into [justfile](justfile) after `gen-python` so it runs automatically.
This has not been implemented yet — the four affected tests are currently
left failing in CI as visible markers.

All four would pass against the LinkML schema and JSON Schema; only the
generated Python dataclass crashes.

## Related fix already applied

`AuthenticationActivityIdEnum` and `DnsActivityActivityIdEnum` previously also
lacked the `UNKNOWN` and `OTHER` permissible values — that part was an OCSF
generator omission, not a LinkML bug, and has been fixed in
[`tools/generate_ocsf_schema.py`](tools/generate_ocsf_schema.py) by merging
the dictionary-level base codes into class-scoped enum overrides. See the
README "Class-scoped enum overrides extend (rather than replace) the
dictionary base" section.
